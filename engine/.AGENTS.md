# Engine Module: Core Game Logic

This directory contains the high-performance Go implementation of the Brass: Birmingham game rules, state management, and Reinforcement Learning (RL) observation generation.

## Contents & Behavior

- **`game.go` / `board.go` / `state.go`**: Core game loop and state structures. Manages players, board setup, and turn sequence.
- **`actions.go`**: Implementation of all player actions (Build, Network, Develop, Sell, Loan, Scout). Includes validation and execution logic.
- **`env.go`**: Provides an OpenAI Gym-like interface (`Reset`, `Step`) for the game, specifically tailored for RL agents.
- **`observation.go`**: Handles the conversion of the current game state into flat float32 tensors for model input. Includes remapping for ego-centric views (First Player perspective).
- **`scoring.go`**: Implements era-transition scoring (Canal to Rail) and final Victory Point (VP) calculation.
- **`network.go` / `resources.go`**: Complex logic for coal/iron resource sourcing and network connectivity (connected-to-market/source checks).
- **`industry_tiles.json` / `map_graph.json`**: Static data for industry stats and map connectivity.
- **`cshared/`**: CGO export layer for building the shared library used by Python.

## Design Patterns

- **Action Registry**: Actions are registered and indexed to provide a consistent discrete action space for RL (approx. 1500 base actions * 8 card slots = 12,000 possibilities, though most are masked).
- **Masking**: High-performance bit-masking for valid actions is implemented to guide the agent during training.
- **Ego-Centric Observations**: The observation tensor is always built from the perspective of the "Active Player," ensuring the model learns a generalized strategy regardless of its player index.

## Key Linkages

- **Consumed By**: `python/` (via direct C-bindings) and `server/` (via API).
- **Traces**: During evaluation, the engine's full state is exported as JSON "traces" for visualization in the dashboard.
