# Server Module: Engine-to-Python Bridge

This directory contains the Go HTTP server that exposes the game engine's functionality to external clients, primarily the Python RL environment.

## Contents & Behavior

- **`main.go`**: The entry point for the API server. Implements a high-performance HTTP gateway to the `engine`.
- **`brass_server.exe`**: Pre-built Windows binary for fast startup in training pipelines.

## API Features

- **Environment Pooling**: Manages multiple concurrent game instances using an ID system (`/envs`).
- **High-Speed Serialization**:
    - **Tensors**: Observations are serialized as binary `float32` (little-endian) and base64-encoded. This is significantly faster than JSON float arrays for Python's `numpy` to ingest.
    - **Masks**: Valid action masks are bit-packed before base64 encoding to reduce payload size.
- **RESTful Endpoints**:
    - `POST /envs`: Create a new environment.
    - `POST /envs/{id}/reset`: Reset the game state.
    - `POST /envs/{id}/step`: Execute an action and return reward/done/observation.
    - `GET /health`: Metadata about observation and action space sizes.

## Design Patterns

- **Concurrency**: Use of `sync.RWMutex` and atomic counters to handle parallel requests from `SubprocVecEnv` in Python.
- **Minimal Logic**: The server is a "thin" wrapper; almost all game-specific logic resides in the `engine` module.

## Key Linkages

- **Depends On**: `engine/` (core logic).
- **Consumed By**: `python/brass_env/` (the Python side of the bridge).
